# Brownfield Existing System — Step-by-Step Example

This example shows how to use the planning skills when the team is **changing an existing system**, not inventing something from scratch.

Scenario:

> A team already has an admin user list page with basic filters. They want to add saved URL state and live search without breaking the current workflow.

The point of this example is to show:
- how to use `CURRENT`
- how to shape a change against an existing baseline
- how to breadboard existing and new affordances together
- how to reason about an existing system before proposing changes

## Files in this example

- `00-source-notes.md` — messy request notes and current-system pain points
- `01-frame.md` — frame for the work
- `02-shaping.md` — shaping with `CURRENT` plus a proposed direction
- `03-breadboard.md` — breadboard combining existing and new behavior
- `04-kickoff.md` — optional handoff doc

## What this example demonstrates

- `CURRENT` is useful when the team is modifying an existing system rather than starting clean.
- The new shape should be compared against the current baseline, not evaluated in isolation.
- Breadboarding can include both current affordances and proposed changes in one story.
- Brownfield work often benefits from existing-system mapping before implementation.
